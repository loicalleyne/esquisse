# GLOSSARY.md — esquisse

Canonical vocabulary for the Esquisse framework. Use these terms exactly —
never invent alternatives for concepts that already have names here.

---

| Term | Definition |
|------|-----------|
| **Esquisse** | The framework itself. A pure-markdown project initialization and AI-agent workflow system. |
| **Phase** | A numbered development stage (`P0`, `P1`, `P2`, …). Each phase has a defined purpose, a set of tasks, and a gate checklist. |
| **Phase Gate** | The set of criteria that must all be true before a phase is declared complete and work on the next phase begins. Verified by `scripts/gate-check.sh`. |
| **Task Document** | A markdown file in `docs/tasks/P{n}-{nnn}-{slug}.md` describing a single, sessionable unit of work. Contains Goal, In Scope, Out of Scope, Files, Acceptance Criteria, and Session Notes. |
| **Skill** | A `SKILL.md` file in `skills/{name}/` containing YAML frontmatter and workflow steps. Auto-triggered by VS Code Copilot Chat or Crush when the user's message matches the `description:` field. |
| **Trigger** | The phrases in a skill's `description:` frontmatter field that cause VS Code Copilot Chat or Crush to load and execute the skill. |
| **Trigger Test** | A manual QA specification in `tests/triggers/{skill-name}.md` documenting exact trigger phrases, expected behavior steps, and pass/fail criteria. |
| **Agent** | A `.agent.md` file in `.github/agents/` (or `~/.copilot/agents/`) that defines a VS Code Copilot Chat sub-agent. Has `name:`, `model:`, `tools:`, and body prompt. Dispatched by `runSubagent()`. |
| **EsquissePlan** | The VS Code Copilot Chat planning agent defined in `.github/agents/EsquissePlan.agent.md`. Decomposes approved specs into task documents and runs adversarial review before handing off to implementation. |
| **Adversarial Review** | A multi-model plan review process. A hostile reviewer (different model from the planner) applies the 7-attack protocol to find gaps, contradictions, and hallucinations in a plan before implementation begins. |
| **7-Attack Protocol** | The seven review attacks applied in adversarial review: False Assumptions, Edge Case Injection, Security Adversary, Logic Contradiction Finder, Context Blindness Check, Failure Mode Analysis, Hallucination Check. |
| **Rotation Slot** | The adversarial reviewer selected by `slot = iteration % 3`. Slot 0 = GPT-4.1 (`Adversarial-r0`), slot 1 = Claude Opus (`Adversarial-r1`), slot 2 = GPT-4o (`Adversarial-r2`). |
| **State File** | `.adversarial/{plan-slug}.json` — records the current review iteration, last model used, and last verdict for a given plan. Schema in `SCHEMAS.md §8`. |
| **Verdict** | The outcome of an adversarial review round: `PASSED`, `CONDITIONAL`, or `FAILED`. |
| **Init** | Running `scripts/init.sh` to bootstrap a new or existing project with Esquisse's document structure and install skills globally. |
| **Adopt** | Using the `adopt-project` skill to add Esquisse framework documents to an existing project that already has source code. |
| **Gate Check** | Running `scripts/gate-check.sh {phase}` to validate all phase gate criteria. |
| **Tool-Name Translation** | The `sed` substitution pass applied by `scripts/init.sh` when installing skills to Crush's skill directory. Replaces VS Code tool names (`runSubagent`, `run_in_terminal`, etc.) with Crush equivalents (`agent`, `bash`, etc.). |
| **CRUSH.md** | A Crush context file generated by `scripts/init.sh` in the project root (after P1-002). Loaded by Crush at session start as part of the system prompt. Equivalent to `bof-bootstrap.instructions.md` for VS Code. |
| **crush-models.md** | A reference file in `skills/adversarial-review/crush-models.md` listing the model strings, rotation table, and security invariant for Crush-mode adversarial dispatch. |
| **esquisse-mcp** | The optional Go MCP stdio server (`esquisse-mcp/`) defined in P1-004. Exposes `adversarial_review` and `gate_review` as MCP tools callable by any MCP client. |
| **AGENTS.md** | The project constitution for AI agents. The first file an agent reads for any project. Contains invariants, conventions, common mistakes, build commands, and security boundaries. |
| **ONBOARDING.md** | Agent orientation file. Defines read order, mental model, key files map, and data flow. Read after AGENTS.md. |
| **GLOSSARY.md** | This file. Canonical domain vocabulary. Agents must use these terms — never invent alternatives. |
| **llms.txt** | Concise public API index, LLM-optimized. |
| **llms-full.txt** | Full LLM-optimized API reference. |
| **ADR** | Architecture Decision Record. Documents a significant design decision, its context, options considered, and the rationale for the choice made. Lives in `docs/adr/ADR-{nnnn}-{slug}.md`. |
| **ROADMAP.md** | Phase-level plan document in `docs/planning/ROADMAP.md`. Tracks current phase, task table, gate checklist, and upcoming phases. |
| **NEXT_STEPS.md** | Current session state in `docs/planning/NEXT_STEPS.md`. Updated at the end of each work session to capture open items, blockers, and the last action taken. |
| **Sessionable** | A task that can be completed in a single AI coding session. Esquisse requires each task to be sessionable: ≤10 files modified, clear acceptance criteria, no open design questions. |
| **P0** | The foundation phase of any project. Establishes the project scaffold: module, CI, core types, interfaces, AGENTS.md. No production features yet. |
| **Stub** | An incomplete function body left as `panic("TODO P{n}-{nnn}: ...")` when a session must end before the task is complete. Stubs are caught by `gate-check.sh`. |
| **Dogfood** | Applying Esquisse's own framework to the Esquisse repository itself — AGENTS.md, ONBOARDING.md, GLOSSARY.md, phase-gated task docs, trigger tests, ROADMAP. |
| **`planning_context`** | A DuckDB table in `code_ast.duckdb` that stores pinned AST symbol snapshots (one row per symbol per task) created by EsquissePlan during Step 2c. Consumed by `implement-task` Step 3b (drift detection) and Step 4a (orientation). |
| **symbol snapshot** | A single row in `planning_context` recording a symbol's name, file path, line numbers, and signature as captured at planning time. |
| **drift detection** | The process of comparing `planning_context` line numbers against the current `ast` table to identify symbols that have moved since planning. Performed at Step 3b of `implement-task`. Advisory only — never blocks implementation. |
